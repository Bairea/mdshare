# Android Markdown 分享卡片工具设计稿

## 1. 背景

国外 AI 问答产品的内容分享，通常存在两个问题：

- 直接分享到微信不方便，甚至需要翻墙或经过多次中转。
- 复制 Markdown 后，现有工具生成的图片经常在表格和代码块场景下显示效果较差，尤其容易出现表格列被截断、代码块在手机宽度下难以阅读的问题。

本产品的目标不是做一个通用 Markdown 编辑器，而是做一个面向 Android 的信息分享工具：将 AI 问答、笔记或总结类 Markdown 内容，快速生成适合微信传播的高质量图片。

核心判断是：

- 产品本质是信息分享，而不是设计创作。
- 功能实现和最终显示效果优先于可配置性。
- 先保证表格和代码块显示质量，再逐步增强入口能力和视觉风格。

## 2. 产品目标

### 2.1 目标用户

- 需要将 AI 回答、知识总结、笔记内容分享到微信的用户。
- 使用 Android 手机，希望减少从复制内容到生成分享图之间的操作步骤的用户。

### 2.2 核心价值

- 将 Markdown 一键转换为适合分享的信息卡片图片。
- 在手机宽度场景下，优先保证表格完整显示、代码块可读。
- 从系统分享或粘贴到出图，流程尽量短。

### 2.3 产品定位

这是一个轻卡片风的信息分享工具，而不是海报设计器。

产品应该具备：

- 轻度卡片美化感
- 强信息传达能力
- 低操作成本

产品不追求：

- 大量模板
- 重装饰风格
- 复杂排版编辑能力

## 3. MVP 范围

### 3.1 必做范围

- Android 端原生应用
- 支持系统 `分享至应用`
- 支持手动粘贴 Markdown
- 提供编辑页、预览页、结果页
- 支持 Markdown 渲染为单张长图
- 支持表格完整显示
- 支持代码块语法高亮
- 支持保存相册和系统分享

### 3.2 不进入 MVP 的能力

- 云端渲染
- 多图分页导出
- 模板市场
- 历史记录和账号体系
- 微信小程序或公众号集成
- 强依赖 ROM 生态的服务卡片能力

### 3.3 MVP 成功标准

- 用户从分享内容到出图，原则上可在 1 分钟内完成。
- 常见 AI 回答内容不需要复杂调整即可直接生成。
- 表格不丢列，代码块不明显损失可读性。
- 导出的图片适合发到微信群、朋友圈或私聊。

## 4. 关键产品原则

### 4.1 信息优先

视觉设计为信息传播服务，不能为了“更好看”而损伤内容完整性。

### 4.2 先完整后精致

优先级顺序如下：

1. 表格完整显示
2. 代码块可读
3. 标题和正文结构清晰
4. 轻卡片风格统一
5. 主题和样式扩展

### 4.3 少步骤

默认主路径应保持为：

1. 从系统分享进入
2. 轻微编辑或直接确认
3. 预览
4. 生成图片
5. 保存或分享

### 4.4 轻卡片风

默认视觉方向采用“稍微有一点卡片美化感”的知识卡片风，而不是营销海报风。

保留的视觉特征：

- 浅色背景
- 圆角主卡片
- 清晰留白
- 标题层级明显
- 表格和代码块作为独立信息区块

避免的视觉特征：

- 强渐变背景
- 大面积装饰图形
- 强品牌化海报风
- 为了好看牺牲信息密度

## 5. 用户场景与主流程

### 5.1 核心场景

用户在浏览器、AI App 或笔记工具中看到一段值得分享的 Markdown 内容，希望尽快生成一张适合微信发送的图片。

### 5.2 MVP 主流程

1. 用户在外部 App 中复制或分享 Markdown。
2. 用户选择“分享至本应用”。
3. 应用直接进入编辑页，并自动填充内容。
4. 用户可直接点击“生成图片”，也可先点“预览”检查效果。
5. 应用本地完成渲染与导出。
6. 用户保存到相册或直接分享。

### 5.3 备用流程

- 用户手动打开应用，进入空白编辑页。
- 用户粘贴 Markdown 后继续主流程。

## 6. 页面设计

### 6.1 编辑页

这是主页面，也是最核心的产品页面。

目标：

- 接住系统分享内容
- 允许最低成本的人工调整
- 快速进入预览或生成

页面结构建议：

- 顶部：返回、页面标题、更多操作
- 内容区：Markdown 文本输入区
- 辅助操作：粘贴、清空
- 底部主操作：预览、生成图片

设计约束：

- 不做复杂格式工具栏
- 不做富文本编辑
- 不做多模板选择入口

### 6.2 预览页

预览页的定位是“结果检查页”，不是“样式编辑器”。

用户主要确认三件事：

- 标题结构是否清晰
- 表格是否完整显示
- 代码块是否仍可阅读

建议操作：

- 返回编辑
- 确认生成

### 6.3 结果页

结果页是完成态页面。

建议结构：

- 图片预览区域
- 主操作：保存相册
- 主操作：立即分享
- 次操作：重新编辑

结果页不承担风格配置职责，避免打断主路径。

## 7. 渲染策略

### 7.1 总体策略

采用固定分享画布，而不是简单截取手机界面。

建议输出规格：

- 宽度固定为 `1080px`
- 高度根据内容自适应
- 以微信分享效果为目标进行样式控制

### 7.2 标题与正文

- 优先使用 Markdown 一级标题作为主标题。
- 若无一级标题，可使用首段内容生成兜底标题。
- 正文以清晰阅读为先，不做复杂多栏布局。

### 7.3 表格渲染规则

表格是本产品的关键能力，必须单独定义渲染规则，而不能只依赖默认网页样式。

#### 原则

- 优先保证所有列完整显示。
- 列少时优先美观。
- 列多时优先完整。

#### 低复杂度表格

适用场景：

- 列数少
- 每列文本长度适中

策略：

- 按内容自然分配列宽
- 不强制把列平均拉满整行
- 保证字体大小和留白合理

目标是避免“只有几个字，但单元格特别宽，整张图大面积空白”的情况。

#### 高复杂度表格

适用场景：

- 列数多
- 文本长度差异大
- 超过手机分享图常规阅读宽度

策略：

- 启用紧凑表格模式
- 允许表格整体缩放
- 允许单元格内容换行
- 适当降低字号和 padding
- 固定表头样式，提高识别度

明确约束：

- 不允许默认只显示前几列
- 宁可整体缩小，也不裁掉后续列

### 7.4 代码块渲染规则

代码块是第二个关键能力。

#### 原则

- 必须有语法高亮
- 必须针对手机分享图宽度进行优化
- 必须优先保证整体可读

#### 规则

- 代码块字号默认小于正文
- 采用更紧凑的行高和内边距
- 使用深色背景或高对比样式，与正文区分
- 超长行优先自动换行
- 若仍过宽，则进一步降低字号

目标是让代码块接近“桌面端约 100 词阅读宽度”的效果，但仍适应手机长图场景。

## 8. 视觉风格规范

### 8.1 风格关键词

- 轻卡片感
- 知识分享
- 清晰层级
- 高信息密度

### 8.2 默认视觉结构

- 整图浅背景
- 主卡片作为内容容器
- 顶部保留轻标签区，例如“AI 摘要”或“内容整理”
- 标题区更突出
- 正文稳定
- 表格和代码块以子卡片方式嵌入

### 8.3 风格边界

允许：

- 圆角
- 轻阴影
- 柔和配色
- 稳定间距系统

不允许：

- 海报化视觉
- 插画主导
- 过度装饰
- 模板驱动的重视觉差异

## 9. 技术方案

### 9.1 推荐技术栈

- 开发语言：`Kotlin`
- UI 框架：`Jetpack Compose`
- Markdown 解析：`flexmark-java`
- 渲染载体：`WebView`
- 代码高亮：`highlight.js`
- 图片保存：`MediaStore`
- 系统分享：Android `Intent`

### 9.2 技术路线判断

推荐使用“原生外壳 + WebView 渲染导出”的方案。

原因：

- Android 原生更适合接系统分享、快捷入口和后续桌面能力。
- WebView 更适合处理表格、代码块、高亮和复杂排版。
- 纯本地方案更适合 MVP，对隐私、性能和稳定性更友好。

不推荐作为 MVP 首选的方案：

- 全原生手写富文本渲染：成本高，表格与代码块适配风险大。
- 纯前端壳应用：系统入口和 Android 原生能力接入较弱。
- 云端渲染：增加后端复杂度和网络依赖，不符合当前最小可用目标。

## 10. 模块拆分

### 10.1 ShareReceiver

- 接收 `ACTION_SEND` / `ACTION_PROCESS_TEXT`
- 识别 Markdown 或普通文本
- 将内容传递给编辑页

### 10.2 EditorViewModel

- 管理原始 Markdown 内容
- 管理标题提取与界面状态
- 驱动预览和生成流程

### 10.3 MarkdownPipeline

- Markdown 转 HTML
- 注入统一模板
- 注入 CSS 样式和本地高亮资源

### 10.4 RenderEngine

- 本地 WebView 加载 HTML
- 监听内容渲染完成
- 计算最终高度
- 导出 Bitmap 或 PNG

### 10.5 ExportModule

- 保存到系统相册
- 调起系统分享

### 10.6 ThemeConfig

- 管理默认视觉 token
- 管理标题、正文、表格、代码块的样式参数
- 为后续主题扩展预留能力

## 11. 异常处理与降级策略

### 11.1 空内容

- 禁止生成
- 提示用户先粘贴或分享 Markdown

### 11.2 Markdown 解析失败

- 降级为纯文本渲染
- 不丢失用户内容

### 11.3 超宽表格

- 强制进入紧凑表格模式
- 不允许默认裁剪列

### 11.4 超长代码块

- 优先换行
- 必要时缩小字号

### 11.5 超长内容

- 先允许导出超长单图
- 后续版本再演进多图分页

### 11.6 导出失败

- 保留渲染结果和原始文本
- 提供重试机制

## 12. 验证策略

### 12.1 样例集

至少准备以下内容类型：

- 普通段落类 AI 回答
- 少列表格
- 多列表格
- 短代码块
- 长代码块
- 表格与代码块混合内容

### 12.2 关键验证点

- 表格是否完整显示所有列
- 表格在列少时是否避免空洞
- 代码块是否仍可阅读
- 微信分享后图片是否清晰
- 系统分享入口是否稳定

### 12.3 测试策略

- 以人工视觉验证为主
- 以典型样例回归为辅
- 自动化测试聚焦主流程和核心转换逻辑，不堆砌低价值 UI 测试

- 预览页

- App Shortcut

## 15. 最终结论

最小可用方案如下：

- 平台：Android
- 入口：系统分享至应用
- 渲染：纯本地 Markdown 转 HTML 再导出图片
- 输出：单张长图
- 风格：轻卡片感的信息分享图
- 技术栈：`Kotlin + Jetpack Compose + flexmark-java + WebView + highlight.js`

这套方案满足以下目标：

- 以产品思维优先服务信息分享场景
- 在最小成本下解决表格和代码块这两个最关键问题
- 为后续快捷入口、小组件和分页能力保留清晰演进空间
